Fat Harness vs Thin Harness

The architecture that fails vs the architecture that scales. The difference is where you put the intelligence.

Fat Harness (Anti-Pattern)

Thick middleware, thin skills, fragile everything

  • 40+ tool definitions eating half the context window
  • God-tools with 2–5 second MCP round-trips
  • REST API wrappers turning every endpoint into a separate tool
  • Business logic baked into harness code instead of skill files
  • 20,000-line CLAUDE.md that degrades model attention

Thin Harness (The Pattern)

~200 lines of glue, fat skills, clean tools

  • Harness does 4 things: loop, files, context, safety
  • Judgment lives in markdown skill files, not code
  • Purpose-built tools: 100ms Playwright vs 15s Chrome MCP (75x)
  • CLAUDE.md is ~200 lines of pointers, not a novel
  • Skills compound — every one is a permanent upgrade
Left: a fat harness cramming tools and logic into the middleware. Right: a thin harness with clean separation — fat skills above, deterministic tools below.

Configuration Layers

The agent's operating manual isn't one monolith — it's nested layers, each with a different job. Click each layer to see its key insight.

CLAUDE.md ~200 lines
The outer shell. Loaded at session start. Should be ~200 lines of pointers, not a 20K-line manual. Anti-pattern: stuffing everything here until model attention degrades. Correct: a routing table that says "when task X, load document Y first."
Resolvers Intent routing
The middle layer. RESOLVER.md reads the user message and dispatches to the right downstream skill. Developer changes a prompt → resolver loads docs/EVALS.md → eval suite runs → revert if accuracy drops >2%. The necessary evolution beyond flat CLAUDE.md.
Skills 90% of value
The innermost layer, where the real work happens. Markdown procedures encoding judgment, process, and domain knowledge. Cross-platform (Claude Code, Cursor, Gemini CLI, Codex). They compound: every skill is a permanent upgrade that never degrades and runs at 3am.

The Enforcement Spectrum

Instructions are soft constraints — the model can ignore them. Hooks and linters are hard constraints — the build fails whether the model likes it or not. The further right you push, the more reliable the system.

Soft Hard
Instructions
Natural language in the prompt. Model may ignore.
CLAUDE.md
Loaded at session start. High compliance, not guaranteed.
Resolvers
Route to the right context. Structural, not just advisory.
Hooks
Shell commands on events. Code runs, not suggestions.
ESLint Rules
Linter rejects bad patterns. Structurally impossible to ship slop.
CI Failures
Build breaks. No override. The hardest constraint.

The Directional Principle

“Push intelligence UP into skills, push execution DOWN into deterministic tooling, keep the harness THIN.”

— The one sentence that captures the entire architecture

Every decision in harness engineering follows from this. Should this logic be in the harness or in a skill? Skill. Should this calculation be done by the model or by code? Code. Should this enforcement be in the prompt or in a hook? Hook.

The model is the most expensive, least reliable, most capable component. Use it for judgment. Use everything else for everything else.
